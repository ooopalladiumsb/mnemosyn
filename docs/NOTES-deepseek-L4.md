## Design choices

### 1. capabilityId JCS shape + parent-omission rule

`capabilityId(cap) = domainHash(CAPABILITY_V1, canonicalBytes(capabilityJcs(cap)))`. The private
`capabilityJcs` helper builds the canonical object as `{schema_version, vault_did, grantee, scope}`
where `scope = {spaces, actions}` (`spaces` is either the literal `"*"` or the string array). The
reserved `parent` field is **omitted from the JCS object whenever it is `undefined`** — so a v1
capability (which never sets `parent`) and a future capability that explicitly leaves `parent` unset
hash to the SAME `capability_id`. This is the L4 analogue of the L0 `created_at`-omission rule: a
reserved field must not change the id until it is actually used. (restricted-JCS byte-sorts keys, so
the TS field order is cosmetic.)

### 2. Signing surface + pubkey/vault binding in verifyGrant

`issueGrant` Ed25519-signs `canonicalBytes(capabilityJcs(cap))` with the raw 32-byte Vault authority
seed (wrapped to PKCS#8 DER via the same `PKCS8_ED25519_PREFIX` machinery as `LocalSigned`); `proof`
is the lowercase-hex signature, `capability_id` the hex id. The signing surface is the capability's
canonical bytes — NOT the id — mirroring how L1 signs `{root,vaultDid,version}`.

`verifyGrant(grant, authorityPublicKey)` returns a boolean and throws ONLY on a non-32-byte pubkey
(`[CAPABILITY_BAD_PUBKEY_LEN]`), mirroring L1 `verifyReceipt`. It enforces three independent checks,
any of which failing → `false`:
1. **Vault binding:** `vaultDidFromPubkey(authorityPublicKey) === grant.capability.vault_did` — the
   grant must have been issued by the authority that DEFINES this vault (a grant signed by some other
   key, even with a valid signature, is rejected because its vault_did won't match).
2. **Id integrity:** `grant.capability_id === hex(capabilityId(grant.capability))`.
3. **Signature:** Ed25519-verify `grant.proof` over `canonicalBytes(capabilityJcs(capability))`,
   building the verifying key as SPKI DER (`302a300506032b6570032100 || raw32`). Bad hex / wrong
   length / bad signature all return `false` (never throw).

`grantAuthorizes(grant, req)` is the pure scope check (no crypto): `vault_did === req.vaultDid`,
`grantee === req.writerDid`, `req.action ∈ scope.actions`, and space allowed (`scope.spaces === "*"`
or includes `req.space`). Authenticity and scope are deliberately separate functions.

### 3. AuthorizingSpine proves the spine is unmodified

`createAuthorizingSpine({spine, authorityPublicKey})` returns a `Spine`-shaped facade. `append(input,
grant)` enforces, in order: `verifyGrant` → `[COLLECTIVE_BAD_GRANT]`; `grantAuthorizes(grant,
{vaultDid, space, action:"append", writerDid})` → `[COLLECTIVE_UNAUTHORIZED]`; `input.capabilityId
=== grant.capability_id` → `[COLLECTIVE_CAPABILITY_MISMATCH]`; then `return spine.append(input)` —
the UNMODIFIED L0 append. `recallById`/`checkpoint` delegate straight through (reads unguarded in v1).
The "Spine untouched" test runs the same input sequence through `AuthorizingSpine` (with valid grants)
and through a bare `spine.append`, and asserts the resulting `vault_memory_root` is identical — the
facade only GATES, it never alters what the spine commits. Combined with the `git diff` showing zero
edits under `src/spine/`, the frozen-L0 guarantee holds structurally and behaviourally.

### 4. Capability golden

`vectors/collective/golden.json` (fifth sibling), `_status` PRE-NORMATIVE, generated by
`scripts/collective-scenario.ts` (wired into `scripts/generate-vectors.ts`). A FIXED authority seed +
3 FIXED capabilities (varying grantee/scope, incl. a `"*"` scope) pin each `capability_id` (64 hex)
and `proof` (128 hex) — proving the id hashing AND the Ed25519 signature are byte-reproducible
(Ed25519/RFC 8032 is deterministic). The golden test does an exact `deepEqual` against the committed
scenario plus a shape check (64-hex id, 128-hex proof). All values are strings → exact comparison.

the only canonical change (the `CAPABILITY_V1` tag) was the architect's, already present.
