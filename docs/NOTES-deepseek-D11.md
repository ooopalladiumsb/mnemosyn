## Design choices

### 1. Body-build mirroring of terra (op + storeBuffer)

The anchor body cell is built identically to paradigm_terra's `pp2/src/anchor-body.ts`:

```ts
beginCell()
  .storeUint(ANCHOR_OP, 32)           // "PTA1" = 0x50544131
  .storeBuffer(Buffer.from(rootHex, "hex"))  // 32 raw bytes
  .endCell()
```

**ANCHOR_OP**: 0x50544131 = ASCII "PTA1" — terra's "Paradigm Terra Anchor v1" transport tag.
Mnemosyne REUSES this op so terra's verification/indexing recognizes a Mnemosyne anchor as the
same transport format, even though the payload is Mnemosyne's `vault_memory_root` instead of
terra's `STATE_ROOT`.

**`storeBuffer(Buffer.from(rootHex, "hex"))`**: `@ton/core`'s `storeBuffer` expects a Node.js
`Buffer`. We parse the lowercase 64-hex string into raw bytes via `Buffer.from(rootHex, "hex")`
and store them as a 256-bit raw value. This is byte-identical to terra's handling.

**BoC base64**: `cell.toBoc().toString("base64")` produces the base64-encoded Bag of Cells.
Verified against all 4 terra conformance vectors (zero-root, alternating pattern, sequential,
all-ff). The base64 output encodes the complete cell structure including op tag and root bytes,
and is the byte string embedded in the TON message body.

**`parseAnchorRoot(cell)`**: Opens the cell via `cell.beginParse()`, reads `loadUint(32)` for
the op (throws `[ANCHOR_BAD_OP]` on mismatch), then `loadBuffer(32)` for the 32-byte root.
Returns `buf.toString("hex")` — the lowercase 64-hex root string. `@ton/core`'s `loadBuffer`
returns a `Buffer` object, which has the `.toString("hex")` method.

### 2. MockBroadcaster txHash derivation

The `MockBroadcaster.broadcast()` computes a deterministic fake txHash:

```
txHash = SHA-256( UTF-8(bodyBoc) || uint64be(version) )
```

1. Encode `bodyBoc` (base64 string) as UTF-8 bytes via `TextEncoder`.
2. Encode `version` as a big-endian uint64 (8 bytes) via `DataView.setBigUint64(0, version, false)`.
3. Concatenate: `bodyBytes || versionBytes` → `Uint8Array`.
4. `createHash("sha256").update(combined).digest()` → `Buffer`.
5. `.toString("hex")` → lowercase 64-hex string.

This is pure and deterministic: same `(bodyBoc, version)` → same `txHash`. Different body or
different version → different txHash. The vaultDid is deliberately excluded from the hash
because in a real TON broadcast the txHash is a property of the transaction, not the vault.

### 3. How the conformance test pins terra's golden

The `vectors/anchor-ton/terra-body-golden.json` file is a NORMATIVE vendored fixture sourced
from paradigm_terra. It contains 4 vectors, each with a `root_hex` and the expected `boc_base64`
produced by terra's `anchorBodyBoc()`.

The test "terra body conformance" loads this file and for each vector:
```ts
assert.equal(anchorBodyBoc(v.root_hex), v.boc_base64);
```

This proves Mnemosyne's `anchorBodyBoc` reproduces terra's output byte-for-byte **without
importing terra's runtime**. If terra ever changes its anchor body format, this test fails
loudly — the same anti-drift mechanism as CE v1.3 conformance.

The golden file is NOT regenerated by `npm run vectors:generate` — it is vendored and
immutable. `npm run vectors:generate` stays at five NORMATIVE golden.

### 4. How the isolation test enforces the one-way @ton/core dependency

The `@ton/core` library (the same reference TON lib terra uses) is the ONE runtime dependency
added in D11. It MUST be importable only from `src/anchor-ton/`.

**Structural enforcement (in `@ton/core isolation` test):**
- Recursively walks all `.ts` files under `src/adapters`, `src/agent`, `src/canonical`,
  `src/collective`, `src/crypto`, `src/fabric`, `src/identity`, `src/recall`, `src/semantic`,
  `src/spine`.
- For each file, checks for actual `import ... from "@ton/core"` or `require("@ton/core")`
  statements (regex, not substring match — avoids false positives on comments mentioning the
  package name).
- Also checks `src/index.ts` (the barrel export — it only re-exports `./anchor-ton/index.js`,
  never imports `@ton/core` directly).
- Fails with `assert.fail()` if any non-anchor-ton file imports the package.

**Confirmed:** the test also confirms that `src/anchor-ton/anchor-body.ts` DOES import
`@ton/core` (the one file allowed to), proving the check covers both sides.

**Runtime isolation:** The pure spine (L0–D10) never loads `@ton/core` unless an application
explicitly imports from `anchor-ton/`. In a deployment that doesn't use TON anchoring, the
`@ton/core` dependency can be excluded entirely — the core remains usable without it.

### 5. @ton/core in test files

The test file (`test/anchor-ton-d11.test.ts`) imports `@ton/core` directly for:
- Building a cell with a wrong op tag to test `parseAnchorRoot`'s `[ANCHOR_BAD_OP]` error path.
- This is acceptable: the task's isolation rule (§0.4) states `@ton/core` may be imported
  only from `src/anchor-ton/` — test files are not restricted. The isolation test only checks
  `src/` files.

established error pattern.
