# Mnemosyne D13.2 — Persistent SpineStore (FileSpineStore)

**Version:** v0.1-draft · **Status:** DRAFT · **Plan:** `docs/PLAN-D13.2.md`
**Depends on:** L0 (`SpineStore`, `MemoryObject`, `memoryObjectId`, `ZERO_HASH_HEX`).

A durable, on-disk `SpineStore` so a vault's memory outlives the process. Drop-in for `MemSpineStore`.
Completes the durable-vault triple: `FileSpineStore` (objects + heads) + `LocalCAS` (ciphertext) +
`LocalSigned({dir})` (anchor chain).

## 1. The `SpineStore` it implements (frozen L0 interface)
```
putObject(obj): Promise<void>                       // store obj; advance its (vault, space) head/count
getObject(vaultDid, objectId): Promise<MemoryObject | null>
spaceCount(vaultDid, space): Promise<number | bigint>   // ZERO if none
spaceHeadHash(vaultDid, space): Promise<string>         // ZERO_HASH_HEX if empty
listSpaces(vaultDid): Promise<readonly string[]>
```

## 2. On-disk layout (`node:fs`, no new dep)
Under `rootDir`, with `vaultKey`/`spaceKey` = lowercase hex of the UTF-8 bytes of the vault DID / space
(bijective → any string is filesystem-safe):
- `<root>/<vaultKey>/objects/<object_id>.json` — one immutable `MemoryObject`. The key `object_id` is
  `memoryObjectId(obj)` (computed in `putObject`).
- `<root>/<vaultKey>/spaces/<spaceKey>.json` — `{ "head": <object_id | ZERO_HASH_HEX>, "count": "<decimal>" }`.

`putObject(obj)`: compute `object_id = memoryObjectId(obj)`; write the object file (creating dirs);
update the space file for `(obj.vault_did, obj.space)` → `head = object_id`, `count = prev count + 1`.
Immutable: writing an object that already exists is a no-op (same content). Append order is recovered
by the spine via the `prev` chain (the store need only store + serve).

## 3. Faithful round-trip (the load-bearing invariant)
`getObject` MUST return an object that re-hashes to the SAME `object_id` it was stored under —
i.e. numeric fields (`seqno`, optional `created_at`) round-trip as the SAME INTEGER VALUE. `MemoryObject`
has `bigint`-capable integers; JSON can't hold `bigint`, so serialize them explicitly (e.g. decimal
strings under a documented convention) and restore the integer. `spaceCount` returns the count as a
`bigint`. Consequence (gated): a spine over `FileSpineStore` yields a byte-identical `vault_memory_root`
to one over `MemSpineStore` for the same appends, and a fresh `FileSpineStore` on the same `rootDir`
sees all prior objects + heads.

## 4. Conformance & gates (OFFLINE — no live step)
1. `npm run typecheck` clean.
2. `npm test` — 262 stay green + new D13.2 tests.
3. `npm run test:conformance` unchanged; `npm run vectors:generate` stays five golden.
4. **MemSpineStore equivalence (load-bearing):** run a multi-append, multi-space spine scenario over
   `createSpine({ store: new FileSpineStore(tmpdir), … })` and over `MemSpineStore`; assert the final
   `vault_memory_root`, every `object_id`, and `space_state` are byte-identical.
5. **Restart:** after appends on `FileSpineStore(dir)`, construct a FRESH `FileSpineStore(dir)`; assert
   `getObject`/`spaceCount`/`spaceHeadHash`/`listSpaces` return the prior state, and a spine over the
   fresh store recomputes the same `vault_memory_root`.
6. **Store ops:** `getObject` of an absent id → `null`; `spaceCount`/`spaceHeadHash` of an empty space
   → 0/`ZERO_HASH_HEX`; `listSpaces` returns the distinct spaces; vault/space strings with `:`/`/`/
   unicode are handled (hex keys). Use `node:fs.mkdtemp` temp dirs; clean them up.
7. No new runtime deps; no new domain tag.

## 5. Decision (proposed — ratify before merge)
**D13.2 (persistent SpineStore).** A file-backed `FileSpineStore(rootDir)` implementing the frozen L0
`SpineStore` over `node:fs`: hex-keyed `<vault>/objects/<id>.json` + `<vault>/spaces/<space>.json`,
faithful numeric round-trip so it is a drop-in for `MemSpineStore` (byte-identical root) and survives
restart. No new dep, no encryption-at-rest (blobs are already ciphertext elsewhere), no concurrency/
journaling (noted). Migration-hard surface frozen by D13.2: the `FileSpineStore` class + ctor. The
MemSpineStore-equivalence (§4.4) + restart-recovery (§4.5) are the controlling acceptance criteria.
