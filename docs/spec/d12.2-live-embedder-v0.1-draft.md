# Mnemosyne D12.2 — Live EmbeddingProvider (OpenAI-compatible)

**Version:** v0.1-draft · **Status:** DRAFT · **Plan:** `docs/PLAN-D12.2.md`
**Depends on:** L2/D6 (`EmbeddingProvider`, `Embedding`, `RecallIndex`, `createRecall`).

The first real, semantic embedder: `OpenAICompatEmbedder` over any OpenAI-compatible `/embeddings`
endpoint. Replaces `HashEmbedder` (deterministic, non-semantic) for live recall. Stays out-of-root
(embeddings are owner-private, never hashed/anchored).

---

## 1. Provider-agnostic by config
`OpenAICompatEmbedderConfig = { baseURL, model, apiKey, dimension, fetchImpl? }`. Factories:
`geminiEmbedder(apiKey, model?, dim?)` (Gemini OpenAI-compat, free) and `jinaEmbedder(apiKey, model?,
dim?)` (Jina, free). Switching providers is config. No new runtime dep — built-in `fetch` (injectable).
DeepSeek has no embeddings API, so the embeddings provider is separate from the Brain's.

## 2. embed protocol
`embed(text)`:
1. `POST {baseURL}/embeddings` via `config.fetchImpl ?? fetch`, headers `Authorization: Bearer
   {apiKey}` + `Content-Type: application/json`, body `{ model, input: text }`.
2. Parse `data[0].embedding` (array of numbers) → `Float32Array`.
3. **Validate** `length === config.dimension` → else throw `[EMBED_DIM_MISMATCH]` (the RecallIndex is
   built with the same dimension; `createRecall` already asserts embedder.dimension === index.dimension).
4. **Errors:** non-2xx, fetch throw, or a malformed/empty response → throw `[EMBED_FAILED] <reason>`.
   The `apiKey` MUST NOT appear in any thrown message. (Unlike the Brain, an embedder MAY throw — it
   is a data op, not a user turn; recall/agent resilience is a separate concern.)

## 3. Boundaries (out-of-root)
- Imports ONLY `./embedding.js` types. Does NOT import the spine, key-manager, or recall-index.
- Embeddings are owner-private, derived — never hashed, never anchored, never committed.
- `apiKey` is config-only: never logged, thrown, or returned.
- Non-determinism (live model) is expected → no golden; the testable surface is the deterministic
  parse + dimension/error handling, exercised with an injected `fetchImpl`.

## 4. Conformance & gates (OFFLINE)
1. `npm run typecheck` clean.
2. `npm test` — 228 stay green + new D12.2 tests.
3. `npm run test:conformance` unchanged; `npm run vectors:generate` stays five golden (no D12.2 golden).
4. **Parse/validate (load-bearing):** injected `fetchImpl` → a canned `{data:[{embedding:[…dim
   floats…]}]}` yields a `Float32Array` of the right length & values; a wrong-length vector →
   `[EMBED_DIM_MISMATCH]`; non-2xx / fetch-throw / missing `data` → `[EMBED_FAILED]` (no apiKey in
   the message). Request shape (URL/headers/model/input) asserted via the mock.
5. **Seam (structural):** `openai-compat-embedder.ts` imports only `./embedding.js`. `apiKey` never
   in a thrown message.
6. Factories: `geminiEmbedder`/`jinaEmbedder` set the right baseURL/model/dimension and `name`.
7. No new runtime deps; no new domain tag.

## 4.1 LIVE smoke (operator-gated, PLAN M5)
With the operator's embeddings key (Gemini/Jina free): index several memories via `createRecall(an
OpenAICompatEmbedder + LocalRecallIndex(dim))`, then a semantic query returns the RELEVANT memory
first (not just "all of them"). Recorded in `docs/notes/`. Not part of the offline merge.

## 5. Decision (proposed — ratify before merge)
**D12.2 (live embedder).** A provider-agnostic `OpenAICompatEmbedder` (+ `geminiEmbedder`/
`jinaEmbedder` factories) implementing the `EmbeddingProvider` seam over `/embeddings`, parsing into a
fixed-`dimension` `Float32Array`, throwing `[EMBED_DIM_MISMATCH]`/`[EMBED_FAILED]` on bad output.
Out-of-root, `apiKey` config-only, built-in `fetch` (no new dep), no golden. Migration-hard surface
frozen by D12.2: `OpenAICompatEmbedderConfig`, `OpenAICompatEmbedder`, `geminiEmbedder`, `jinaEmbedder`.
Parse/validate correctness (§4.4) + the seam boundary (§4.5) are the controlling acceptance criteria.
